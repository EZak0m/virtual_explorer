# 仮想探索Mod (Virtual Explorer) 要件定義書 (v2)

## 1. プロジェクト概要とテーマ
- **対象環境**: Minecraft 1.21.1 / NeoForge
- **主目的**: ワールド（チャンク）を実際に生成・肥大化させることなく、エネルギー（FE）とリソース（白紙の地図）を消費して仮想的にアイテムを自動収集する。
- **基本テーマ**: 「地図と探索」
- **ゲームプレイ**: モジュール類の基本クラフト素材に「地図」や「コンパス」などの探索用アイテムを使用。工業系Modでの自動生産ライン構築という遊びを提供する。

## 2. 主要ブロック・アイテム
### 2.1. 仮想探索卓 (Virtual Mapping Table Block)
- **コア機能**: 本Modの中心となる機械（BlockEntity）。
- **ホッパー・パイプ制限**: NeoForgeの `IItemHandler` 機能を利用し、外部からのアクセスを厳格に管理。
  - **入力許可**: FE、白紙の地図（専用スロットのみ）。
  - **出力許可**: 出力インベントリのアイテムのみ搬出可能。
  - **アクセス不可**: モジュールスロット、ユーティリティスロットのアイテムはプレイヤーの手動操作でのみ出し入れ可能。
- **スタンバイ状態**: 出力スロットが満杯になった場合、FEと地図の消費を停止し、探索プロセスを一時停止する。

### 2.2. モジュール・アイテム群
- **指定モジュール**: 地下探索、地上探索、ネザー探索、エンド探索など。
- **構造物フィルタ**: 空中右クリックでGUIを開き、対象とする構造物をオン/オフできるツール。
- **無限化モジュール**: ネザースター等を素材とするエンドコンテンツ。構造物を「探索済み」にせず、永遠に同じ構造物を抽出し続ける。
- **アップグレード**: スピードアップグレード（処理速度UP、FE・地図の消費量増加）。
- **ユーティリティ**: 幸運、シルクタッチ、自然生成物取得のオン/オフ切り替えモジュール。

## 3. 動作ロジック
### 3.1. 自然生成物の探索（データ解析型）
- 現在のバイオームデータ（Biome -> PlacedFeature）を参照し、生成されるはずだったブロックリストを取得。
- モジュール条件に合致するブロックをランダム選出し、LootTableを参照してアイテム化。Mod追加鉱石等も自動対応。
- **FakePlayerの利用**: 幸運などのツール効果を正しく適用するため、内部に「偽のプレイヤー」を保持し、LootContextを適切に構築してドロップ判定を行う。

### 3.2. 構造物の探索とワールド保護
- **索敵の最適化**: Locate処理は非常に重いため「最初の1回」および「次の構造物へ移動するタイミング」のみ非同期（または負荷分散）で実行。
- **解析**: 到達後、対象構造物の `StructureTemplate`（NBT）をメモリ上で読み込み、チェスト総数と外壁ブロックを算出。
- **WSD (World Saved Data) 記録**: 探索済みとなった構造物がある「チャンク座標」をディメンションごとに分けてセーブデータに記録（データ肥大化防止のため領域ではなくチャンク単位で記録）。
- **チェスト削除**: プレイヤーが該当箇所を訪れチャンクが新規生成される際（`ChunkEvent.Load`）、WSDと照合し、探索済みであればチェストを空（または空気に置換）にする。

## 4. Config（設定ファイル）の充実
柔軟な調整ができるよう、`serverconfig`と`clientconfig`を用意する。

### 4.1. Server Config (virtualexplorer-server.toml)
- **エネルギー設定**: 基本FE容量 (デフォルト: 1,000,000 FE)、1Tickあたりの基本FE消費量 (デフォルト: 100 FE)
- **リソース消費設定**: 1チャンク移動に必要な「白紙の地図」の枚数 (デフォルト: 1)
- **アップグレード係数**: スピードアップグレード1枚あたりの速度上昇倍率と、エネルギー消費増加倍率。
- **バランス・負荷設定**: 探索処理を行う間隔（Tick数）、構造物の外壁ブロックを取得する確率（%）、ワールド生成時のチェスト削除機能のオン/オフ、ブラックリスト。

### 4.2. Client Config (virtualexplorer-client.toml)
- GUIのテーマカラー変更や、稼働中のパーティクル表示の有無。

## 5. Lang（言語・ローカライズファイル）の充実
- `en_us.json` と `ja_jp.json` を整備。
- **GUIテキスト**: 「探索中…」「インベントリ満杯」「白紙の地図が不足しています」「FEが不足しています」など。
- **ツールチップ (Tooltip)**: モジュールアイテムにカーソルを合わせ、Shiftキー押下で詳細仕様を表示。
- **JEI / EMI連携**: 「Information」タブに対応させ、ゲーム内で使い方や仕様を読めるようにする。

## 6. アーキテクチャと設計方針（拡張性とテスト容易性）
本プロジェクトでは、将来の機能追加（新しいモジュールやアルゴリズム）や仕様変更に柔軟に対応できるよう、以下の設計方針を採用します。

### 6.1. 関数の細分化と純粋関数 (Pure Functions) の活用
- 複雑な処理（例：探索ロジック、アイテム抽選、エネルギー消費計算など）を巨大な一つのメソッドに記述せず、小さな関数に分割します。
- 状態を持たない純粋関数を多用することで、同じ入力に対して常に同じ出力を返すようにし、テストの予測可能性を高めます。

### 6.2. ロジックとMinecraft APIの分離
- Minecraftのワールド操作やNBTデータの読み書きなど、Minecraft固有のAPIに依存する層と、純粋なデータ処理（確率計算、リストフィルタリング等）を行う層を明確に分離します。
- これにより、Minecraftクライアント/サーバー環境を起動しなくても、JUnit等のフレームワークを用いて高速に単体テスト（Unit Test）を実行できるようにします。

### 6.3. インターフェース主導設計
- 各モジュールやアップグレードアイテムの振る舞いを定義するインターフェース（例: `IExplorerModule`, `IUpgradeItem`）を作成します。
- 新しいモジュールを追加する際は、既存のコアコードを変更するのではなく、新しいインターフェースの実装クラスを追加するだけで済むようにします（開放閉鎖の原則 - OCP）。

### 6.4. 依存性の注入とテスト用モック
- イベントシステムやデータ永続化（WSD）などのコンポーネントは、密結合を避ける設計とします。これにより、テスト実行時に「ダミーのワールド」や「ダミーのインベントリ」を注入しやすくなり、エッジケースのテストが容易になります。
